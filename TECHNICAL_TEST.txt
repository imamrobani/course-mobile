Mobile Frontend Take-Home Test: React Native App with Auth, State Management, 
Favorites, and Profile 
Overview 
The goal of this exercise is to evaluate your ability to build a modern mobile application 
using React Native or Expo. 
You will build a small course app that demonstrates not only UI implementation, but 
also how you handle: 
• API integration 
• authentication 
• shared state management 
• navigation 
• user interactions 
• mobile user experience across multiple screens 
The app should include: 
• Login flow using MockAPI 
• Course browsing 
• Course detail screen 
• Comments and likes 
• Favorites / saved courses 
• Editable profile screen 
• Centralized state management 
We are primarily evaluating: 
• Code quality and readability 
• Architecture and maintainability 
• State management 
• API integration 
• Navigation and mobile UX 
• Ability to model realistic app behavior

Main Objective 
Build a mobile course app where a user can: 
• Log in using a user account from MockAPI 
• Persist the session locally 
• Browse and filter courses 
• View course details 
• Like and unlike comments 
• Add comments as the logged-in user 
• Save courses to favorites 

---PAGE---

• Access saved courses from a dedicated screen 
• View and edit their profile 
 
API Requirements 
1. MockAPI Authentication 
You must use MockAPI for login-related data. 
Create a users resource in MockAPI. 
Example user structure 
User 
- id 
- name 
- email 
- password 
- avatar 
- bio 
Example record 
{ 
  "id":"1" , 
  "name":"John Doe" , 
  "email":"john@example.com" , 
  "password":"123456" , 
  "avatar":"https://... " , 
  "bio":"Mobile learner" 
} 
Login behavior 
There is no real authentication backend required, so you can simulate login like this: 
• Fetch users from MockAPI 
• Match user by email 

---PAGE---

• Validate the entered password against the returned user record 
• If credentials are correct:  
o mark the user as logged in 
o persist the logged-in user in local storage 
• If credentials are incorrect:  
o show an error message 
Minimum login requirements 
• Login form with:  
o email 
o password 
• Validation for empty fields 
• Loading state while checking credentials 
• Error state for invalid login 
• Persist session locally using AsyncStorage or similar 
• Auto-login if a valid user session is found on app reopen 
• Logout functionality 
Notes 
You do not need to implement signup unless you want to. 
The goal is to evaluate: 
• API integration 
• async handling 
• form validation 
• loading and error states 
• session persistence

Required Features 

---PAGE---

1. Splash / App Loading Screen 
Create an initial loading screen that checks whether a user session already exists. 
Expected behavior 
• On app start, check local storage for logged-in user 
• If user exists:  
o go directly to the authenticated flow 
• If not:  
o go to Login screen 
This does not need to be visually complex, but it should be part of the app flow. 
 
2. Login Screen 
Create a login screen connected to MockAPI. 
Requirements 
• Input fields:  
o email 
o password 
• Validate required fields 
• Show loading state during login request 
• Show invalid credentials error 
• On success:  
o save user session locally 
o navigate to the main app flow 
UX expectations 
• Proper keyboard handling 
• Disabled submit button while loading 
• Clear error messages 
 
3. Home Screen 

---PAGE---

Create a course list screen. 
Each course card must display: 
• Image 
• Title 
• Short description 
• Category 
• Rating 
• Author or level 
Requirements 
• Search by title 
• Category filter 
• Empty state when no results 
• Mobile-friendly list layout 
• Tap course to navigate to detail screen 
Additional requirement 
Each course card or detail screen must allow the user to: 
• save / unsave the course 
• visually see whether the course is already saved 
This can be shown with a bookmark, heart, or save icon. 
 
4. Course Detail Screen 
The detail screen must display: 
• Course image 
• Title 
• Author 
• Category 
• Duration 
• Level 

---PAGE---

• Full description or content 
• Back navigation 
• Save / unsave action 
Requirements 
• A user must be able to save a course from this screen 
• The saved state should update immediately in the UI 
 
5. Comments Section 
Display comments for the selected course. 
Each comment must show: 
• User name 
• Message 
• Created date 
• Likes count 
• Like button 
Requirements 
• Logged-in user can add a comment 
• New comments appear immediately 
• Like / unlike updates immediately 
• Shared state should manage comments and likes cleanly 
We want to see whether you can model this beyond isolated local component state. 
 
6. Favorites / Saved Courses Screen 
Create a dedicated screen where the user can access saved courses. 
Requirements 
• Show all courses the user has saved 
• Each saved item should show useful course information 
• Tapping a saved course opens its detail page 

---PAGE---

• Empty state if there are no saved courses 
• User can remove a course from favorites from this screen or from the detail 
screen 
Expected behavior 
If a user saves a course: 
• it should appear in the Favorites / Saved Courses screen 
If a user unsaves a course: 
• it should disappear from that screen 
What we want to evaluate 
• Shared state design 
• Data flow between screens 
• Navigation consistency 
• Feature completeness 
 
7. Profile Screen 
Create a profile screen for the logged-in user. 
The profile screen must display: 
• Avatar 
• Name 
• Email 
• Bio or short description 
Requirements 
• Show current logged-in user information 
• Include an Edit Profile action 
• Include Logout 
 
8. Edit Profile Screen 
Add a screen where the user can edit their profile. 

---PAGE---

Editable fields 
At minimum: 
• name 
• avatar 
• bio 
You may allow email editing too, but it is optional. 
Requirements 
• Pre-fill the form with current user data 
• Save changes to app state 
• Persist changes locally 
• Return to profile screen with updated data visible immediately 
Bonus 
If you want, you may also update the user in MockAPI, but this is optional unless you 
explicitly choose to support it. 
What we want to evaluate 
• Form handling 
• Editable state 
• Shared state updates 
• Persistence 
• User flow between view and edit screens 
 
9. State Management

Use it for at least: 
• Authentication state
